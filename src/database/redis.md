## 一、5 种基本数据类型

### 命令前缀规律

命令前缀就是类型缩写，记住前缀就记住了命令：

| 类型 | 常用命令 | 记忆方式 |
|------|---------|---------|
| String | `SET` / `GET` | 无前缀 |
| Hash | `HSET` / `HGET` | 前缀 `H` |
| List | `LPUSH` / `RPOP` | 前缀 `L`/`R`，代表左右两端 |
| Set | `SADD` / `SMEMBERS` | 前缀 `S` |
| ZSet | `ZADD` / `ZRANGE` | 前缀 `Z` |

---

### 1. String — 单个值

```
SET user:name "张三"
GET user:name           → "张三"
INCR order:count        → 原子自增，适合计数器
EXPIRE user:token 3600  → 设置过期时间（秒）
```

**典型场景**：缓存、计数器、Token 存储

---

### 2. Hash — 对象（多个字段）

value 是一个 Map，适合存对象：

```
HSET user:1 name "张三" age 27 city "北京"
HGET user:1 name        → "张三"
HGETALL user:1          → 所有字段
```

**命令解读**：`HGET user:1 name` = 先找 Redis key `user:1`，再找 Hash 内部 field `name`

内存结构：

```
"user:1" → {
    "name": "张三",
    "age":  "27",
    "city": "北京"
}
```

**vs String 方案：**

```
String：SET user:1 '{"name":"张三","age":27}'
  改年龄 → GET → 反序列化 → 改字段 → 序列化 → SET（搬运整个对象）

Hash：HSET user:1 age 28
  改年龄 → 直接改单个字段，其他字段不受影响
```

**典型场景**：存用户信息、商品详情

---

### 3. List — 有序队列

有序、可重复，支持两端操作：

```
LPUSH queue "任务A"   → 从左插入
LPUSH queue "任务B"   → 从左插入，现在是 [B, A]
RPOP  queue           → 从右取出 → 返回 "任务A"
```

**典型场景**：消息队列（左进右出）、最新动态列表

---

### 4. Set — 无序集合，不重复

```
SADD user:1:fans "小明" "小红" "小李"
SADD user:2:fans "小红" "小张"

SINTER user:1:fans user:2:fans  → 交集 → "小红"（共同粉丝）
```

**典型场景**：去重、共同关注、抽奖

---

### 5. ZSet — 有序集合，带分数

比 Set 多了一个 score，Redis 自动按 score 排序：

```
ZADD rank 100 "张三"
ZADD rank 300 "李四"
ZADD rank 200 "王五"

ZRANGE rank 0 -1 WITHSCORES
→ 张三:100, 王五:200, 李四:300（从低到高）
```

**命令解读**：`ZADD rank 100 "张三"` = key 是 `rank`，score 是 `100`，member 是 `张三`

**典型场景**：排行榜、延迟队列

---

### 命名约定

`user:1`、`order:1001` 中的冒号只是**开发者约定**，Redis 不识别，不按约定也不会报错。价值在于 RedisInsight 等可视化工具会识别冒号，自动显示树形结构。

## 二、过期策略

Redis 设置过期时间：

```
SET token "abc123" EX 3600   → 1小时后过期
```

时间到了，Redis 通过两种方式组合删除：

### 惰性删除

不主动删，等访问时才检查：

```
GET token
  → 过期了 → 删掉，返回 nil
  → 未过期 → 正常返回
```

缺点：没人访问的 key 永远占内存。

### 定期删除

每隔一段时间**随机抽取**一批 key 检查，过期的删掉。随机抽取是因为全扫太慢会阻塞请求。

缺点：随机抽，抽不到的漏网。

### 为什么组合？

```
只用惰性 → 没人访问的 key 永远占内存
只用定期 → 抽不到的 key 也永远占内存
组合使用 → 大部分过期 key 都能被清理
```

---

## 三、内存淘汰策略

过期 key 清理有漏网之鱼，内存迟早会满。内存满了，Redis 按配置策略**主动踢掉一些 key**。

两个维度：

```
踢谁？
  allkeys  = 所有 key 都能踢
  volatile = 只踢有过期时间的 key

怎么踢？
  lru    = 最久没被访问的先踢
  lfu    = 访问次数最少的先踢
  random = 随机踢
  ttl    = 快过期的先踢
```

| 策略 | 含义 |
|------|------|
| `allkeys-lru` | 所有key中淘汰最久未使用的 ★最常用 |
| `allkeys-lfu` | 所有key中淘汰使用频率最低的 |
| `allkeys-random` | 所有key中随机淘汰 |
| `volatile-lru` | 有过期时间的key中淘汰最久未使用的 |
| `volatile-lfu` | 有过期时间的key中淘汰使用频率最低的 |
| `volatile-random` | 有过期时间的key中随机淘汰 |
| `volatile-ttl` | 有过期时间的key中优先淘汰快过期的 |
| `noeviction` | 不淘汰，内存满直接报错（默认） |

**选型：** 缓存场景首选 `allkeys-lru`，被踢了可以从数据库重新加载。

**两者配套关系：**

```
过期策略   → 处理"到期的 key"
内存淘汰   → 处理"内存不够用"
```

## 四、持久化

Redis 是内存数据库，重启数据就没了。持久化把内存数据存到磁盘，重启后能恢复。

### RDB — 快照（定期拍照存档）

**RDB = Redis Database**，把某一时刻的全部数据存成二进制文件（`dump.rdb`）。

```
# 配置文件触发条件（满足任一条件就触发）
save 900 1      → 900秒内有1个key改动
save 300 10     → 300秒内有10个key改动
save 60 10000   → 60秒内有10000个key改动
```

- 优点：文件小，恢复速度快
- 缺点：两次快照之间宕机，这段时间数据全丢

```
10:00 拍了快照
10:04 写了很多数据
10:05 宕机
→ 10:00~10:05 这5分钟的数据全没了
```

---

### AOF — 追加日志（记流水账）

**AOF = Append Only File**，把每一条写命令追加记录到日志文件（`appendonly.aof`），重启时重放所有命令恢复数据。

```
SET user:name "张三"   → 写入日志
HSET user:1 age 27    → 写入日志
DEL user:2            → 写入日志
```

三种刷盘策略：

```
always   → 每条命令立刻写磁盘（最安全，性能最差）
everysec → 每秒写一次 ★推荐，最多丢1秒数据
no       → 由操作系统决定（性能最好，可能丢更多）
```

- 优点：数据丢失少，最多丢 1 秒
- 缺点：文件越来越大，恢复慢

**AOF 重写**解决文件膨胀：

```
对 count 执行了1000次 INCR → AOF 记录了1000条命令
重写后 → 直接记 SET count 1000（1条命令搞定）
```

---

### 混合模式 — 生产环境推荐

Redis 4.0 后支持，结合两者优点：

```
重启恢复：
  先加载 RDB 快照（快速恢复大部分数据）
  再重放 RDB 之后的 AOF 增量日志（补上快照后的数据）
  = 快速 + 完整
```

### 对比

| | RDB | AOF |
|--|-----|-----|
| 记忆方式 | 定期拍照存档 | 记流水账 |
| 数据丢失 | 多（分钟级） | 少（最多1秒） |
| 恢复速度 | 快 | 慢 |
| 文件大小 | 小 | 大 |

## 五、缓存穿透 / 击穿 / 雪崩

### 缓存穿透 — 查不存在的数据

```
用户查 id=99999 的商品
→ Redis 没有 → 去数据库查
→ 数据库也没有 → 返回空
→ 下次还查 → 还是穿透到数据库
```

缓存完全没起作用，恶意攻击者用不存在的 id 狂刷，数据库直接被打垮。

**解决方案：**

```
方案1：缓存空值
  数据库查不到 → 往 Redis 存空值，设短过期时间
  SET product:99999 "null" EX 60
  缺点：攻击者每次换不同的非法id，缓存空值拦不住，还占 Redis 内存

方案2：布隆过滤器（推荐）
  提前把所有合法 id 存入布隆过滤器
  查询前先问布隆过滤器：这个 id 存在吗？
  → 不存在 → 直接拦截，连 Redis 都不查
```

**生产标准做法：布隆过滤器（拦截绝大多数）+ 缓存空值（兜底误判漏网之鱼）**

---

### 缓存击穿 — 一个热点 key 过期

```
"双十一主会场"这个 key 每秒被查几万次
突然过期
→ 几万个请求同时发现 Redis 没有
→ 同时去数据库查
→ 数据库瞬间压垮
```

**解决方案：**

```
方案1：热点 key 永不过期
  不设过期时间，后台定时任务异步更新数据

方案2：互斥锁 + double check
  key 过期后只让一个线程去重建缓存，其他线程等待
  拿到锁之后先再查一次 Redis（double check）
  → 有值了直接返回，不用再查数据库
```

互斥锁核心代码：

```java
String value = redis.get(key);
if (value != null) return value;

lock.lock();
try {
    // double check：拿到锁后再查一次
    value = redis.get(key);
    if (value != null) return value;

    // 确认没有，才查数据库
    value = db.query(key);
    redis.set(key, value);
    return value;
} finally {
    lock.unlock();
}
```

---

### 缓存雪崩 — 大量 key 同时过期

```
批量导入缓存，过期时间全部设成1小时
1小时后，这批 key 同时过期
→ 大量请求同时打到数据库 → 数据库崩了
```

与击穿的区别：

```
击穿 → 一个热点 key 过期
雪崩 → 大量 key 同时过期
```

**解决方案：**

```
方案1：过期时间加随机值
  redis.set(key, value, 3600 + random.nextInt(300))
  让过期时间分散，不会同时失效

方案2：Redis 高可用（基础设施层面）
  主从复制 + 哨兵/集群，防止 Redis 宕机导致雪崩

方案3：熔断降级
  缓存没有 → 不查数据库，直接返回默认值
  保护数据库不被继续压垮
```

---

### 三大问题汇总

```
穿透 → 查不存在的数据    → 布隆过滤器 + 缓存空值
击穿 → 一个热点key过期   → 永不过期 + 互斥锁
雪崩 → 大量key同时过期   → 过期时间加随机值 + 高可用
```

## 六、分布式锁

### 为什么需要分布式锁？

`synchronized`、`ReentrantLock` 只在单个 JVM 进程内有效：

```
服务器A 的线程A 加了锁
服务器B 的线程B 完全不知道
→ 两个线程同时执行，数据错乱
```

多台服务器部署时需要一个**所有服务器都认的第三方锁**，Redis 天然适合（高性能 + 支持过期时间）。

---

### 演进过程

#### 第一版：SETNX

```
SETNX lock:order "1"
→ 返回1 → 设置成功，拿到锁
→ 返回0 → 已有人持有，等待
```

**问题：线程拿到锁后宕机，锁永远不释放（死锁）**

---

#### 第二版：加过期时间

```
SET lock:order "1" EX 30 NX
→ NX = 不存在才设置
→ EX 30 = 30秒后自动过期，防止死锁
```

**问题：业务执行超过30秒，锁自动过期，其他线程趁虚而入**

---

#### 第三版：看门狗自动续期（Redisson）

```
线程A 拿到锁
后台看门狗线程每隔10秒检查：
  线程A 还在执行 → 自动续期（重置为30秒）
  线程A 结束了  → 不续期，等自然过期
```

**问题：服务器宕机，看门狗也挂了，锁过期后服务重启，僵尸线程执行完删掉了别人的锁**

---

#### 第四版：UUID 验证（最终方案）

```java
// 加锁时存入唯一标识
SET lock:order "线程A的UUID" EX 30 NX

// 释放时先验证是不是自己的锁
if (redis.get("lock:order").equals(自己的UUID)) {
    redis.del("lock:order");  // 是自己的才删
}
// 不是自己的 → 直接跳过，不处理
```

---

### 完整流程

```
加锁：SET key uuid EX 30 NX
        ↓
业务执行（看门狗自动续期）
        ↓
释放：先验证 uuid 是自己的，再 DEL
```

---

### 为什么选 Redis 做分布式锁？

```
天然支持过期时间  → 防死锁
性能极高          → 单线程 + 纯内存，10万+QPS
公司基本都有      → 不需要额外引入组件
```

> 第三方不一定是 Redis，Zookeeper 也能做分布式锁，原理类似。Redis 被广泛使用是因为性能好且公司基本都有。

**生产上直接用 Redisson，它把所有问题都处理好了，不需要自己手写。**

## 七、Redis 为什么快？

面试必考，三点记住：

```
1. 纯内存操作    → 不涉及磁盘IO，速度极快
2. 单线程无锁    → 无锁竞争，无上下文切换开销
3. IO多路复用    → 一个线程处理大量客户端连接
```

**IO 多路复用：**

```
传统方式：1000个客户端 → 需要1000个线程
IO多路复用：一个线程同时监听1000个连接
           哪个连接有数据来了，就处理哪个
```

> Redis 不是完全单线程，处理客户端请求是单线程，后台持久化、删除过期key等是多线程。
