# MySQL 索引原理

## 一、为什么需要索引？

业务量小的时候，数据库全表扫描速度感知不明显，建不建索引差别不大。

随着业务增长，数据到了千万级，用户根据某个字段查询数据，没有索引的情况下全表扫描极慢，直接影响用户体验。业务不关心你的技术实现，只关心"为什么搜个东西这么久"。

**索引的本质**：给数据建一个目录，查询时先查目录定位数据，而不是一行一行扫描。

---

## 二、为什么用 B+树，不用哈希表？

### 哈希表的问题

哈希表查单个值是 O(1)，看起来比 B+树更快，但有一个致命缺陷：**不支持范围查询**。

哈希函数的设计目的是把数据**打散均匀分布**，18 和 19 这两个相邻的数字，在哈希表里的位置可能相差十万八千里，完全没有顺序关系。

```sql
WHERE age > 18 AND age < 30
```

执行这个查询时，数据库不知道 18~30 之间有哪些值，只能把哈希表从头扫到尾，看每个值是否满足条件——这和全表扫描没有任何区别。

而实际业务中范围查询非常常见：

```sql
WHERE create_time > '2024-01-01'
WHERE price BETWEEN 100 AND 500
WHERE age > 18
```

**结论：哈希表只能精确匹配，不支持范围查询，不适合做数据库索引。**

## 三、B+树结构原理

### 基本结构

B+树是一棵**多叉树**（不是二叉树），分两种节点：

```
非叶子节点（目录页）：只存 key，用来导航，不存真实数据
叶子节点（数据页） ：存 key + 完整数据，所有叶子节点用链表串联
```

示意图（7个学生数据）：

```
                    ┌────────┐
                    │   10   │           ← 根节点，只存 key
                    └────────┘
                    /         \
                   /           \
          ┌──────────┐     ┌──────────┐
          │   1 | 5  │     │ 15 | 20  │  ← 非叶子节点，只存 key
          └──────────┘     └──────────┘
           / |   |  \        /  |    \
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│id=1  │ │id=5  │ │id=10 │ │id=15 │ │id=20 │ │id=25 │ │id=30 │
│小明  │→│小红  │→│小李  │→│小张  │→│小王  │→│小赵  │→│小孙  │
└──────┘ └──────┘ └──────┘ └──────┘ └──────┘ └──────┘ └──────┘
                    链表从左到右有序 →
```

### 节点和叉数的关系

节点里有 N 个 key，就有 N+1 个子节点：

```
[15 | 30]  ← 2个key，把数轴分成3段

< 15      15~30      > 30
  ↓          ↓          ↓
左子树     中子树      右子树
```

### 为什么是"矮胖"树？

MySQL 每次读磁盘以**页**为单位，默认一页 **16KB**。B+树每个节点对应一个磁盘页，节点尽量塞满一页：

```
一个非叶子节点 = 一个16KB的页
每个key约8字节，指针约6字节
16KB ÷ 14字节 ≈ 1170 个key → 1171叉

三层B+树能存：1170 × 1170 × 叶子数据 ≈ 2000万行
查任意一行：只需要 3 次磁盘IO
```

对比二叉树存2000万数据需要约24层，B+树只需3~4层，查询次数极少。

**叉数由磁盘页大小决定，不是数据量。数据量影响的是树的层数。**

### 查询过程演示

精确查询 `WHERE id = 20`：

```
根节点：20 >= 10 → 去右边
右节点：20 在 15~20 之间 → 去中间
叶子节点：找到 id=20，小王 ✅（共3次IO）
```

范围查询 `WHERE id >= 10 AND id <= 25`：

```
定位到 id=10 的叶子节点
顺着链表往右扫：
  id=10 小李 ✅ → id=15 小张 ✅ → id=20 小王 ✅ → id=25 小赵 ✅
  id=30 > 25，停止，返回4条数据
```

**叶子节点链表保证了范围查询能顺着扫，这是B+树支持范围查询的关键。**

## 四、聚簇索引 vs 非聚簇索引

### 聚簇索引

MySQL 的**主键索引**就是聚簇索引，数据和索引存在一起：

```
叶子节点：
┌─────────────────────────────────┐
│ id=1 | name=小明 | age=18 | ... │  ← 完整的数据行
└─────────────────────────────────┘
```

查 `WHERE id = 1`：走主键B+树 → 叶子节点直接拿到完整数据 → 一次查询搞定

### 非聚簇索引

自己建的普通索引（如给 age 建索引），叶子节点只存索引键和主键：

```
叶子节点：
┌──────────────┐
│ age=18 | id=1│  ← 只存 age 和主键 id，没有完整数据
└──────────────┘
```

查 `WHERE age = 18 AND name = '小明'`：

```
走 age 的B+树 → 找到 id=1
→ 再拿 id=1 去主键B+树查一次 → 拿到完整数据（包括name）
```

**这个"再查一次主键索引"的过程叫做回表，有额外开销。**

---

## 五、覆盖索引

如果查询的字段，索引里都有，就不需要回表：

```sql
-- 给 age 建了索引，叶子节点存 age + id
SELECT id, age FROM student WHERE age = 18
-- SELECT 需要的字段只有 id 和 age，索引里全有 → 不需要回表
```

这叫**覆盖索引**，是 SQL 优化的常用手段。

`SELECT *` 是大忌——多出来的字段索引里没有，必然回表。

## 六、联合索引 + 最左前缀原则

### 联合索引

```sql
ALTER TABLE student ADD INDEX idx_age_name (age, name);
```

B+树的排序规则：先按 age 排，age 相同再按 name 排：

```
叶子节点（有序）：
age=18, name=小李
age=18, name=小明
age=18, name=小红
age=20, name=小王
...
```

### 最左前缀原则

查询条件必须从最左边的字段开始，不能跳过：

```sql
-- 索引是 (age, name)

WHERE age = 18                    → ✅ 命中索引
WHERE age = 18 AND name = '小明'  → ✅ 命中索引
WHERE name = '小明'               → ❌ 未命中（跳过了 age）
WHERE name = '小明' AND age = 18  → ✅ 命中（MySQL自动调整顺序）
```

### 为什么必须从最左开始？

`WHERE name = '小明'` 单独查时：

```
name 在整棵树里是无序的
age=18 里有小明，age=20 里也可能有小明
没有规律可循 → 只能全表扫描
```

`WHERE age = 18` 先查时：

```
age 有序 → 直接定位到 age=18 的范围
在 age=18 范围内，name 是有序的 → 继续精确定位
```

### 联合索引的字段顺序设计

根据实际查询场景设计，把区分度高、常用的字段放前面：

```sql
-- 高频查询：app_id + api_name + 时间范围
ALTER TABLE api_log ADD INDEX idx_query (app_id, api_name, create_time);

-- 查询时：
WHERE app_id = 'xxx' AND api_name = 'yyy' AND create_time > '2024-01-01'
→ 层层过滤，效率最高
```

## 七、字符串索引的排序规则

字符串按**字符编码值**排序，本质和数字大小比较相同：

```
'小明' → 转成字节码 → E5 B0 8F E6 98 8E
'小红' → 转成字节码 → E5 B0 8F E7 BA A2

从第一个字节开始逐字节比较：
前3字节都是 E5 B0 8F（都是"小"字）
第4字节 E6 < E7 → '小明' < '小红'
```

这也解释了：
- `LIKE '小%'` → 能走索引（前缀固定，字节码有序）
- `LIKE '%明'` → 走不了索引（前缀不固定，没有规律）

---

## 八、索引设计原则

```
1. 根据高频查询场景建索引，不要无脑建
2. 写SQL时尽量命中索引，避免 SELECT *
3. 联合索引字段顺序要结合查询条件设计
4. 索引建多了写操作变慢（每次写入要维护B+树）
5. 千万级数据 + 合理索引，查询性能完全稳定
```

## 九、面试高频问答

### Q1：为什么 MySQL 用 B+树而不用哈希表？

**答：** 哈希表不支持范围查询。哈希函数把数据打散存储，丢失了数据的大小顺序关系。执行 `WHERE age > 18` 时，数据库不知道比18大的值在哪些桶里，只能全表扫描，和没有索引效果一样。B+树叶子节点按顺序排列并用链表串联，范围查询直接定位起点顺着扫就行。

---

### Q2：什么是回表？怎么避免？

**答：** 走普通索引查询时，叶子节点只存索引键和主键，如果 SELECT 的字段不在索引里，需要拿主键再去主键索引查一次完整数据，这个过程叫回表。

避免方式：使用覆盖索引，让 SELECT 的字段都包含在索引中，或者建联合索引把需要的字段都加进去，不用回表直接返回。

---

### Q3：联合索引的最左前缀原则是什么？

**答：** 联合索引的 B+树按照字段顺序排序，先按第一个字段排，第一个字段相同再按第二个排。查询时必须从最左边的字段开始用，不能跳过，否则后面的字段在树里是无序的，无法利用索引。

比如索引是 `(age, name)`，只用 `WHERE name = '小明'` 查询时，name 在整棵树里无序，只能全表扫描。

---

### Q4：为什么不能无脑建索引？

**答：** 索引不是越多越好。每次 INSERT、UPDATE、DELETE 操作，MySQL 除了修改数据本身，还要同步维护所有相关索引的 B+树（插入节点、重新平衡）。索引越多，写操作越慢。应该根据高频查询场景有针对性地建索引。
