# synchronized 锁升级

## 核心：Java 对象头（Mark Word）

每个 Java 对象在堆中都有一个**对象头**，64 位 JVM 下 Mark Word 占 8 字节，记录了锁状态信息：

```
|-----------------------------------------------------|
|          Mark Word（64 bit）                        |
|-----------------------------------------------------|
| 无锁    | hashCode(31) | age(4) | 0 | 01 |
| 偏向锁  | threadID(54) | epoch | age(4) | 1 | 01 |
| 轻量级锁 | 指向栈帧Lock Record的指针(62)      | 00 |
| 重量级锁 | 指向Monitor对象的指针(62)          | 10 |
| GC标记  |                                    | 11 |
|-----------------------------------------------------|
```

后两位是**锁标志位**，是判断锁状态的关键。

---

## 四种锁状态及升级过程

### 1. 无锁（01）
对象刚创建，无竞争，Mark Word 存 hashCode。

### 2. 偏向锁（01，偏向标志位=1）

**触发条件**：第一个线程访问同步块。

**原理**：
- 将当前线程 ID 用 CAS 写入 Mark Word
- 下次同一线程再进入，只需比较线程 ID，**无需任何 CAS 操作**
- 适用场景：**只有一个线程访问**的场景（单线程重复加锁开销极低）

**撤销时机**：有第二个线程竞争时，升级为轻量级锁（需 STW 到安全点撤销）

> JDK 15 起默认关闭偏向锁（-XX:+UseBiasedLocking），JDK 21 彻底移除

### 3. 轻量级锁（00）

**触发条件**：多线程交替执行（无真正并发竞争）。

**原理**：
1. 线程在**栈帧**中创建 Lock Record，存储 Mark Word 的拷贝
2. CAS 将 Mark Word 替换为指向 Lock Record 的指针
3. CAS 成功 → 获得锁；CAS 失败 → 自旋等待

**升级时机**：
- 自旋超过阈值（默认 10 次，或自适应自旋）
- 有第三个线程来竞争
→ 升级为重量级锁

**适用场景**：线程交替执行，竞争时间极短

### 4. 重量级锁（10）

**原理**：
- 依赖操作系统 Mutex Lock（互斥量）
- 未竞争到锁的线程进入**阻塞状态**（从用户态切换到内核态）
- 涉及线程上下文切换，**开销最大**

**Monitor 结构**：
```
Monitor {
    _owner      → 持有锁的线程
    _EntryList  → 等待获取锁的线程队列（BLOCKED）
    _WaitSet    → 调用 wait() 后等待的线程队列（WAITING）
}
```

---

## 升级流程图

```
对象创建
   ↓
无锁（hashCode）
   ↓ 第一个线程访问
偏向锁（写入 threadID，无 CAS）
   ↓ 第二个线程竞争
轻量级锁（CAS + 自旋，不阻塞线程）
   ↓ 自旋失败 / 第三个线程竞争
重量级锁（OS Mutex，线程阻塞）
```

> **锁只能升级，不能降级**（重量级锁无法退回轻量级锁）

## 高频面试题

### Q1：synchronized 和 ReentrantLock 的区别？

| 对比项 | synchronized | ReentrantLock |
|--------|-------------|---------------|
| 实现层级 | JVM 内置，字节码层面 | JDK，基于 AQS |
| 锁释放 | 自动释放（出块/异常） | 必须手动 `unlock()` |
| 可中断 | 不可中断 | `lockInterruptibly()` 支持 |
| 公平锁 | 不支持 | 支持 `new ReentrantLock(true)` |
| 条件变量 | 一个（wait/notify） | 多个（Condition） |
| 性能 | JDK 6 后差距不大 | 高竞争场景略优 |

### Q2：偏向锁为什么在 JDK 15 被废弃？

维护成本高，且现代应用多线程并发普遍，偏向锁的撤销（STW）反而成为开销，
得不偿失。

### Q3：自旋锁为什么不能无限自旋？

自旋期间线程仍占用 CPU，若持有锁的线程执行时间长，自旋只会浪费 CPU 资源，
不如阻塞挂起。JVM 的**自适应自旋**会根据上次自旋的成功率动态调整自旋次数。

### Q4：结合你的项目 —— API 百万调用场景下用 synchronized 合适吗？

不合适。高并发场景下 synchronized 的重量级锁阻塞开销大，
推荐：
- 无状态操作 → 不加锁
- 计数/统计 → `AtomicLong` / `LongAdder`
- 共享资源保护 → `ReentrantLock` + 超时机制，避免无限等待
- 限流 → Redis + Lua（跨进程）

## 基础概念补充（通俗理解）

### 🔐 锁究竟锁住的是什么？

**一句话：锁住的是"一段代码"的入口，保证同一时刻只有一个线程能执行它。**

#### 为什么需要锁？

```java
// 两个线程同时执行 count++，会出错！
int count = 0;
count++;  // 实际分三步：读 count → 加 1 → 写回
          // 两个线程同时做这三步，会互相覆盖
```

`count++` 看起来一行，CPU 实际要分三步执行。两个线程同时做，就会互相覆盖，最终少加了。

**加锁就是为了让这三步不被打断：**

```java
synchronized(lockObject) {
    count++;  // 同一时刻只有一个线程能进来执行
}
```

#### 锁记录在哪里？

`synchronized(lockObject)` 括号里的 `lockObject` 是一个普通 Java 对象，锁的状态信息记在这个对象的**对象头**里（Mark Word 那张"小纸条"）。

```java
Object lock = new Object();  // 随便一个对象都能当锁

synchronized(lock) {
    // 线程A 进来，把状态写进 lock 的对象头
    // 线程B 想进来，发现 lock 对象头显示"有人"，等着
}
```

#### 用生活场景理解

> 想象一个**公共厕所**，只有一个坑位（共享资源）

```
多个人（线程）都想进去（执行 count++）

没有锁 → 两个人挤进去 → 乱套（数据错误）

加了锁 → 里面的人插上插销（锁住对象头）
        → 外面的人看到插销 → 等
        → 里面的人出来 → 解锁 → 下一个人进
```

**锁的对象（`lockObject`）就是那个插销**，插销贴在门上（对象头 Mark Word），记录着"是谁插的、有没有人排队"。

> 注意：README 中的"无锁（01）"是对象头的**出厂默认状态**，是锁升级的起点，不是说不用加锁。只要你写了 `synchronized`，线程一进来就立刻从"无锁"状态开始往上升级。

### 🔄 CAS 是什么？

**CAS = Compare And Swap（比较并交换），是 CPU 提供的一条原子指令。**

#### 三个参数：内存地址 V、期望值 A、新值 B

> "如果内存里的值还是我期望的 A，就把它换成 B；如果不是 A 了，说明别人动过，我就放弃。"

**整个"比较 + 替换"是一步完成的，CPU 保证中间不会被打断。**

#### 用生活场景理解

想象黑板上写着数字 `5`，两个人同时想把它改成 `6`：

```
张三：我记得黑板上是 5，我要改成 6
      → 走过去一看，确实还是 5 → 改成 6 ✅ 成功

李四：我也记得是 5，我也要改成 6
      → 走过去一看，已经是 6 了，不是我期望的 5 → 改失败 ❌
      → 李四可以选择：重新读一遍，再试一次（自旋重试）
```

**这就是 CAS：先比较，相同才替换，保证只有一个人能成功。**

#### 对应到轻量级锁

```
// 对象头的 Mark Word 初始值 = "无锁标记"

线程A 来了：
  CAS("期望是无锁标记", "换成指向我的 Lock Record 的指针")
  → 成功，进入同步块 ✅

线程B 来了（此时对象头已经是线程A的指针了）：
  CAS("期望是无锁标记", "换成指向我的 Lock Record 的指针")
  → 失败，对象头不是"无锁标记"了 ❌
  → 开始自旋（原地转圈重试）
```

#### CAS vs 普通赋值

| | 普通赋值 `a = 1` | CAS |
|--|-----------------|-----|
| 原子性 | ❌ 读和写是两步，可能被打断 | ✅ 比较+替换一步完成 |
| 线程安全 | ❌ | ✅ |
| 是否阻塞 | 不涉及 | **不阻塞**，失败了就重试 |

> **一句话记住**：CAS = "我以为是 A，如果还是 A 就换成 B，否则不动"，整个操作 CPU 保证原子性。这就是为什么轻量级锁"不阻塞线程"——它不是"排队等"，而是"一直尝试 CAS 直到成功"，这个不断重试的过程就叫**自旋**。

### 🧩 Mark Word 64位是怎么分配的？

64位就像一排 **64个小格子**，不同锁状态下，这64个格子的用途完全不同。

#### 最后两位永远是"锁标志位"

无论哪种状态，**最右边 2 位**永远固定用来表示锁的类型，JVM 先看这两位来判断当前是什么锁：

```
xx xx xx xx ... [第2位][第1位]
                  ↑
              这两位就是锁标志位
```

| 锁标志位 | 含义 |
|---------|------|
| `01` | 无锁 或 偏向锁（再看偏向标志位区分） |
| `00` | 轻量级锁 |
| `10` | 重量级锁 |
| `11` | GC 标记 |

---

#### 无锁状态（标志位 = `01`，偏向位 = `0`）

```
| 25位(未用) | 31位(hashCode) | 1位(未用) | 4位(GC年龄) | 1位(偏向=0) | 2位(01) |
 ←————————————————————————— 64 位 ——————————————————————————→
```

- **hashCode（31位）**：对象的哈希码，调用 `Object.hashCode()` 时生成并存在这里
- **GC年龄（4位）**：对象经历了几次 GC 还没被回收，最大 15（4位二进制最大值），超过 15 就晋升老年代
- **偏向标志（1位）**：`0` = 没有偏向任何线程
- **锁标志（2位）**：`01`

> 为什么 hashCode 只有31位？因为 Java `hashCode()` 返回 `int`（32位），实际有效位31位。

---

#### 偏向锁状态（标志位 = `01`，偏向位 = `1`）

```
| 54位(线程ID) | 2位(epoch) | 1位(未用) | 4位(GC年龄) | 1位(偏向=1) | 2位(01) |
 ←————————————————————————— 64 位 ——————————————————————————→
```

- **threadID（54位）**：记录当前持有偏向的线程ID，下次该线程进来对比一下就能直接进
- **epoch（2位）**：偏向锁的批量撤销时间戳，可理解为"版本号"，用于判断偏向是否过期
- **偏向标志（1位）**：`1` = 已偏向某个线程

> 注意：无锁和偏向锁的锁标志位都是 `01`，JVM 靠**倒数第3位（偏向标志位）**来区分两者。

---

#### 轻量级锁状态（标志位 = `00`）

```
| 62位（指向当前线程栈帧中 Lock Record 的内存地址） | 2位(00) |
 ←————————————————————————— 64 位 ——————————————————————————→
```

- **62位指针**：指向当前持锁线程在自己栈帧里创建的 Lock Record（里面存着原来的 Mark Word 备份）
- 64位系统上内存地址本来就64位，但因为内存对齐，地址最低几位永远是0，所以62位足够表示所有有效地址

---

#### 重量级锁状态（标志位 = `10`）

```
| 62位（指向堆中 Monitor 对象的内存地址） | 2位(10) |
 ←————————————————————————— 64 位 ——————————————————————————→
```

- **62位指针**：指向堆中的 Monitor 对象（里面有 `_owner`、`_EntryList`、`_WaitSet`）

---

#### 一张图看清全貌

```
位编号：  [63 ........... 10][9 ........ 7][6][5 .. 4][3   ][2][1 0]
                                                              ↑    ↑
                                                          偏向位  锁标志位

无锁：   [unused:25][hashCode:31]         [unused:1][age:4][0 ][0 1]
偏向锁： [threadID:54][epoch:2]           [unused:1][age:4][1 ][0 1]
轻量级锁：[ptr_to_LockRecord:62]                            [0 0]
重量级锁：[ptr_to_Monitor:62]                               [1 0]
GC标记： [                  62位空                        ][1 1]
```

#### 一句话总结

> Mark Word 就像一张**多用途表格**，总共64格，最右两格永远是"类型标识"，其余62格根据当前锁状态**复用**：没竞争时存 hashCode，有线程偏向时存线程ID，轻/重量级锁时存指针。

### 🔒 ReentrantLock 是什么？为什么会出现在这里？

#### 先说背景

`synchronized` 是 Java 最早的加锁方式，JVM 内置，用起来简单：

```java
synchronized(lock) {
    // 同步代码
}  // 出了这个块自动解锁
```

但它有几个**硬伤**：
- 抢不到锁就**死等**，没法设超时
- 不能**中途放弃**等锁
- 只有一个"等待室"（`wait/notify`），不能区分不同条件的等待
- 只支持**非公平锁**（谁抢到算谁的，不管等了多久）

于是 JDK 5 推出了 `ReentrantLock`，作为 `synchronized` 的**升级替代品**，解决这些问题。

---

#### ReentrantLock 怎么用？

```java
ReentrantLock lock = new ReentrantLock();

lock.lock();        // 加锁（抢不到就一直等，类似 synchronized）
try {
    // 同步代码
} finally {
    lock.unlock();  // 必须手动解锁！放在 finally 保证一定执行
}
```

> ⚠️ **最大区别**：`synchronized` 自动解锁，`ReentrantLock` 必须手动 `unlock()`，忘了就死锁。

---

#### ReentrantLock 的额外能力

```java
// 1. 可中断等待：等锁时可以被其他线程中断，不用死等
lock.lockInterruptibly();

// 2. 超时等待：最多等 3 秒，抢不到就算了
boolean got = lock.tryLock(3, TimeUnit.SECONDS);

// 3. 公平锁：按先来后到排队，不允许插队
ReentrantLock fairLock = new ReentrantLock(true);

// 4. 多个条件变量：可以区分不同的等待原因
Condition notFull  = lock.newCondition();  // 等"不满"
Condition notEmpty = lock.newCondition();  // 等"不空"
// synchronized 只有一个 wait/notify，没法区分
```

---

#### 为什么面试题放在一起比较？

因为它们**解决同一个问题**（线程安全），只是实现方式不同：

| | `synchronized` | `ReentrantLock` |
|--|----------------|-----------------|
| 出身 | JVM 内置关键字 | JDK 的一个类（基于 AQS） |
| 解锁 | 自动 | 手动 `unlock()` |
| 超时/中断 | ❌ | ✅ |
| 公平锁 | ❌ | ✅ |
| 多条件等待 | ❌（只有 wait/notify） | ✅（多个 Condition） |
| 性能 | JDK 6 后差距不大 | 高竞争场景略好 |

---

#### 什么时候用哪个？

```
简单场景、代码短 → 用 synchronized（简洁，自动释放，不容易出错）

需要超时、可中断、公平性、多条件 → 用 ReentrantLock

高并发场景（API 百万调用）→ 考虑 AtomicXxx / LongAdder，
                            甚至直接 无锁设计（CAS）
```

> **记住**：`ReentrantLock` 不是 `synchronized` 的升级版，而是**功能更丰富的补充**。大多数场景 `synchronized` 够用，别为了用新的而用新的。

### 🏗️ AQS 是什么？（ReentrantLock 的底层基础）

#### 一句话

> **AQS = AbstractQueuedSynchronizer（抽象队列同步器）**，是 JDK 并发包里的一个**基础框架类**，`ReentrantLock`、`CountDownLatch`、`Semaphore` 等都是基于它实现的。

---

#### 用"停车场"来理解 AQS

AQS 就像一个**停车场管理系统**，它维护了两样东西：

```
1. 一个"车位状态"（state，整数）
   - state = 0  → 车位空着，可以进
   - state = 1  → 车位被占了
   - state > 1  → 同一辆车停了多次（可重入）

2. 一条"等待车道"（CLH 队列，一排等待的车）
   - 进不去的车排成一列，按顺序等
   - 车位空了，叫队头的车进来
```

---

#### AQS 的核心结构

```java
// AQS 内部（简化）
public abstract class AbstractQueuedSynchronizer {
    
    private volatile int state;  // 锁状态，0=未锁，1=已锁，>1=重入次数
    
    // 等待队列（双向链表，存放等待的线程）
    // Head → [线程A] ↔ [线程B] ↔ [线程C] → Tail
    
    // 子类实现：尝试加锁（CAS 把 state 从 0 改成 1）
    protected boolean tryAcquire(int arg) { ... }
    
    // 子类实现：尝试解锁（把 state 减1，到0就释放）
    protected boolean tryRelease(int arg) { ... }
}
```

---

#### ReentrantLock 怎么用 AQS？

```
线程A 来抢锁：
  CAS(state: 0 → 1) → 成功 ✅ → 记录"我是持有者"，进入同步块

线程B 来抢锁：
  CAS(state: 0 → 1) → 失败 ❌（state 已经是 1）
  → 把自己包装成节点，加入 AQS 等待队列，挂起

线程A 重入（再次进入同步块）：
  发现持有者是自己 → state 从 1 变成 2（重入计数+1）
  → 这就是"可重入"的意思

线程A 解锁：
  state 从 2 → 1（重入退出一层）
  state 从 1 → 0（完全释放）
  → 唤醒队列里的线程B → 线程B 抢到锁
```

---

#### AQS 为什么重要？

它不只是 `ReentrantLock` 的底层，整个 `java.util.concurrent` 包里大量工具都基于它：

| 工具类 | 用 AQS 干什么 |
|--------|--------------|
| `ReentrantLock` | state=0/1 表示锁空闲/占用 |
| `CountDownLatch` | state 从 N 倒数到 0，到 0 就放行所有等待线程 |
| `Semaphore` | state 表示剩余许可数，为 0 就等待 |
| `ReentrantReadWriteLock` | state 高16位=读锁数，低16位=写锁数 |

> AQS 就是一个**万能的排队框架**，把"排队等待"和"唤醒"的通用逻辑都实现好了，子类只需要实现"怎么判断能不能进"（`tryAcquire`/`tryRelease`）就行。

---

#### 和 synchronized 重量级锁的 Monitor 对比

```
synchronized 的 Monitor：           AQS（ReentrantLock 的底层）：
  _owner = 持有线程                   state = 锁状态（整数）
  _EntryList = 等待获取锁的线程       CLH队列 = 等待获取锁的线程
  _WaitSet = wait()后等待的线程       Condition队列 = await()后等待的线程
  
  区别：Monitor 在 JVM/OS 层实现      AQS 纯 Java 实现，更灵活可控
```
