# synchronized 锁升级

## 核心：Java 对象头（Mark Word）

每个 Java 对象在堆中都有一个**对象头**，64 位 JVM 下 Mark Word 占 8 字节，记录了锁状态信息：

```
|-----------------------------------------------------|
|          Mark Word（64 bit）                        |
|-----------------------------------------------------|
| 无锁    | hashCode(31) | age(4) | 0 | 01 |
| 偏向锁  | threadID(54) | epoch | age(4) | 1 | 01 |
| 轻量级锁 | 指向栈帧Lock Record的指针(62)      | 00 |
| 重量级锁 | 指向Monitor对象的指针(62)          | 10 |
| GC标记  |                                    | 11 |
|-----------------------------------------------------|
```

后两位是**锁标志位**，是判断锁状态的关键。

---

## 四种锁状态及升级过程

### 1. 无锁（01）
对象刚创建，无竞争，Mark Word 存 hashCode。

### 2. 偏向锁（01，偏向标志位=1）

**触发条件**：第一个线程访问同步块。

**原理**：
- 将当前线程 ID 用 CAS 写入 Mark Word
- 下次同一线程再进入，只需比较线程 ID，**无需任何 CAS 操作**
- 适用场景：**只有一个线程访问**的场景（单线程重复加锁开销极低）

**撤销时机**：有第二个线程竞争时，升级为轻量级锁（需 STW 到安全点撤销）

> JDK 15 起默认关闭偏向锁（-XX:+UseBiasedLocking），JDK 21 彻底移除

### 3. 轻量级锁（00）

**触发条件**：多线程交替执行（无真正并发竞争）。

**原理**：
1. 线程在**栈帧**中创建 Lock Record，存储 Mark Word 的拷贝
2. CAS 将 Mark Word 替换为指向 Lock Record 的指针
3. CAS 成功 → 获得锁；CAS 失败 → 自旋等待

**升级时机**：
- 自旋超过阈值（默认 10 次，或自适应自旋）
- 有第三个线程来竞争
→ 升级为重量级锁

**适用场景**：线程交替执行，竞争时间极短

### 4. 重量级锁（10）

**原理**：
- 依赖操作系统 Mutex Lock（互斥量）
- 未竞争到锁的线程进入**阻塞状态**（从用户态切换到内核态）
- 涉及线程上下文切换，**开销最大**

**Monitor 结构**：
```
Monitor {
    _owner      → 持有锁的线程
    _EntryList  → 等待获取锁的线程队列（BLOCKED）
    _WaitSet    → 调用 wait() 后等待的线程队列（WAITING）
}
```

## 升级流程图

```
对象创建
   ↓
无锁（hashCode）
   ↓ 第一个线程访问
偏向锁（写入 threadID，无 CAS）
   ↓ 第二个线程竞争
轻量级锁（CAS + 自旋，不阻塞线程）
   ↓ 自旋失败 / 第三个线程竞争
重量级锁（OS Mutex，线程阻塞）
```

> **锁只能升级，不能降级**（重量级锁无法退回轻量级锁）

## 高频面试题

### Q1：synchronized 和 ReentrantLock 的区别？

| 对比项 | synchronized | ReentrantLock |
|--------|-------------|---------------|
| 实现层级 | JVM 内置，字节码层面 | JDK，基于 AQS |
| 锁释放 | 自动释放（出块/异常） | 必须手动 `unlock()` |
| 可中断 | 不可中断 | `lockInterruptibly()` 支持 |
| 公平锁 | 不支持 | 支持 `new ReentrantLock(true)` |
| 条件变量 | 一个（wait/notify） | 多个（Condition） |
| 性能 | JDK 6 后差距不大 | 高竞争场景略优 |

### Q2：偏向锁为什么在 JDK 15 被废弃？

维护成本高，且现代应用多线程并发普遍，偏向锁的撤销（STW）反而成为开销，
得不偿失。

### Q3：自旋锁为什么不能无限自旋？

自旋期间线程仍占用 CPU，若持有锁的线程执行时间长，自旋只会浪费 CPU 资源，
不如阻塞挂起。JVM 的**自适应自旋**会根据上次自旋的成功率动态调整自旋次数。

### Q4：结合你的项目 —— API 百万调用场景下用 synchronized 合适吗？

不合适。高并发场景下 synchronized 的重量级锁阻塞开销大，
推荐：
- 无状态操作 → 不加锁
- 计数/统计 → `AtomicLong` / `LongAdder`
- 共享资源保护 → `ReentrantLock` + 超时机制，避免无限等待
- 限流 → Redis + Lua（跨进程）
