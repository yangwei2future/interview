## 一、内存结构

JVM 运行时数据区分为两类：**线程私有** 和 **线程共享**。

```
┌─────────────────────────────────────────────┐
│                  JVM 内存                   │
│                                             │
│  线程私有（每个线程独立一份）                  │
│  ┌──────────┐ ┌──────────┐ ┌─────────────┐ │
│  │ 程序计数器 │ │ 虚拟机栈  │ │  本地方法栈  │ │
│  └──────────┘ └──────────┘ └─────────────┘ │
│                                             │
│  线程共享（所有线程共享）                      │
│  ┌──────────────────────┐ ┌──────────────┐ │
│  │         堆           │ │   方法区(元空间)│ │
│  └──────────────────────┘ └──────────────┘ │
└─────────────────────────────────────────────┘
```

### 1. 程序计数器（PC Register）

- 记录当前线程正在执行的字节码指令地址
- 线程切换后恢复执行位置靠它
- **唯一不会 OOM 的区域**（不存储数据，只存地址）

---

### 2. 虚拟机栈（JVM Stack）

- 每次方法调用创建一个**栈帧**，方法返回栈帧出栈
- 栈帧包含：局部变量表、操作数栈、动态链接、返回地址
- **StackOverflowError**：递归太深，栈帧超过栈深度上限（默认 -Xss 512k~1M）

```java
// 典型场景：无终止条件的递归
public void infinite() {
    infinite();  // StackOverflowError
}
```

---

### 3. 堆（Heap）

- **最大的内存区域**，存放所有对象实例和数组
- GC 主要工作区域
- **OOM（OutOfMemoryError: Java heap space）**：对象太多无法回收

分代模型（经典分代 GC 使用）：

```
┌─────────────────────────────────────────────────────┐
│                        堆                           │
│  ┌────────────────────────┐  ┌────────────────────┐ │
│  │        新生代            │  │      老年代          │ │
│  │  ┌───────┐ ┌──┐ ┌──┐  │  │                    │ │
│  │  │ Eden  │ │S0│ │S1│  │  │    长期存活对象       │ │
│  │  └───────┘ └──┘ └──┘  │  │                    │ │
│  │   8:1:1 比例            │  │                    │ │
│  └────────────────────────┘  └────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

- **Eden**：新对象首先分配到 Eden
- **Survivor（S0/S1）**：Minor GC 后存活对象移到 Survivor
- **老年代**：经过多次 GC 存活（年龄阈值默认 15）的对象晋升到老年代

---

### 4. 方法区 / 元空间（Metaspace）

- 存放：类的元数据（类名、方法、字段、常量池）
- JDK 8 之前叫"永久代"（PermGen），JDK 8 改为**元空间**，使用本地内存
- **OOM（OutOfMemoryError: Metaspace）**：类太多（动态生成类、反射滥用）

> **为什么从 PermGen 改为 Metaspace？**
> PermGen 大小固定，容易 OOM；Metaspace 使用本地内存，上限是物理内存，更灵活。

---

### 5. 运行时常量池

- 属于方法区，存放字面量（字符串、数字）和符号引用
- `String.intern()` 把字符串放入常量池

## 二、对象创建过程

`new Object()` 背后 JVM 做了什么：

```
1. 类加载检查   → 检查类是否已加载，没有则先加载
2. 分配内存     → 堆中划出一块内存（指针碰撞 or 空闲列表）
3. 初始化零值   → 将内存区域清零（int→0, ref→null）
4. 设置对象头   → 记录类信息、哈希码、GC 年龄、锁状态
5. 执行 <init> → 调用构造方法，字段赋值
```

**对象头（Object Header）结构：**

```
┌──────────────┬────────────────────┐
│  Mark Word   │   Klass Pointer    │
│ (哈希/锁/GC) │  （指向类元数据）    │
└──────────────┴────────────────────┘
```

Mark Word 根据锁状态存不同数据：
- 无锁：哈希码 + GC 年龄
- 偏向锁：线程 ID
- 轻量级锁：指向栈中锁记录的指针
- 重量级锁：指向 Monitor 对象的指针

## 三、GC 垃圾回收

### 3.1 如何判断对象是否可以回收

**可达性分析（GC Roots）**

从一组 GC Roots 出发，能遍历到的对象是存活的，遍历不到的是垃圾。

GC Roots 包括：
- 虚拟机栈中正在运行方法引用的对象（局部变量）
- 方法区中类静态属性引用的对象
- 方法区中常量引用的对象
- JNI 引用的对象（本地方法）

> **为什么不用引用计数？**
> 循环引用（A 引用 B，B 引用 A）会导致引用计数永远不为 0，对象无法回收。Java 选择可达性分析避免这个问题。

---

### 3.2 四种引用类型

| 引用类型 | 特点 | 场景 |
|---------|------|------|
| 强引用（StrongReference） | 只要引用在，GC 绝不回收 | 普通 `Object obj = new Object()` |
| 软引用（SoftReference） | 内存不足时才回收 | 缓存（内存紧张时自动淘汰） |
| 弱引用（WeakReference） | 下次 GC 就回收，不管内存够不够 | ThreadLocal 的 key |
| 虚引用（PhantomReference） | 随时被回收，拿不到对象 | 跟踪对象被回收，做资源清理 |

---

### 3.3 GC 算法

**标记-清除（Mark-Sweep）**
```
步骤：标记存活对象 → 清除未标记对象
缺点：产生大量内存碎片，大对象可能找不到连续空间
```

**标记-整理（Mark-Compact）**
```
步骤：标记存活对象 → 存活对象移到内存一端 → 清除边界外内存
优点：无碎片
缺点：移动对象开销大，需要暂停用户线程（STW）
```

**复制算法（Copying）**
```
步骤：将内存分两半 → 存活对象复制到另一半 → 清空原来那半
优点：无碎片，速度快
缺点：内存利用率只有 50%（新生代优化为 Eden+S0+S1 的 8:1:1）
```

**新生代用复制算法**（因为每次 GC 绝大多数对象都死了，复制的少，快）

**老年代用标记-整理**（对象大多存活，复制开销大，用整理更合适）

---

### 3.4 分代 GC 触发条件

| GC 类型 | 触发条件 | 回收范围 | STW |
|--------|---------|---------|-----|
| Minor GC | Eden 区空间不足 | 新生代 | 短暂 |
| Major GC | 老年代空间不足 | 老年代 | 较长 |
| Full GC | 老年代不足 / 方法区不足 / 显式 `System.gc()` | 堆 + 方法区 | 最长 |

**对象晋升老年代的条件：**
1. GC 年龄超过阈值（默认 15，`-XX:MaxTenuringThreshold`）
2. 单次 Survivor 区放不下
3. 大对象直接进老年代（`-XX:PretenureSizeThreshold`）

---

### 3.5 常见垃圾收集器

| 收集器 | 作用区域 | 算法 | 特点 |
|-------|---------|------|------|
| Serial | 新生代 | 复制 | 单线程，客户端模式 |
| ParNew | 新生代 | 复制 | 多线程版 Serial，配合 CMS |
| Parallel Scavenge | 新生代 | 复制 | 吞吐量优先 |
| Serial Old | 老年代 | 标记-整理 | Serial 的老年代版 |
| CMS | 老年代 | 标记-清除 | 低延迟，但有碎片 |
| G1 | 全堆 | 复制+标记整理 | 可预测停顿，JDK 9+ 默认 |
| ZGC | 全堆 | 染色指针 | 超低延迟（<10ms），JDK 15+ 生产可用 |

**G1 的特点（重点）：**
- 将堆划分为多个等大的 Region（1~32MB），不再固定分代边界
- 优先回收垃圾最多的 Region（Garbage First 的由来）
- 停顿时间可预测：`-XX:MaxGCPauseMillis=200`（目标值）
- 适合大内存服务端（>4GB）

## 四、类加载机制

### 4.1 类加载流程

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
```

**加载**：读取 .class 文件字节流，创建 Class 对象

**验证**：检查字节码是否符合 JVM 规范（防止恶意字节码）

**准备**：为静态变量**分配内存并赋零值**（int 为 0，非常量）
```java
// 准备阶段：value = 0
// 初始化阶段：value = 123
static int value = 123;

// 常量在准备阶段直接赋值 123
static final int CONST = 123;
```

**解析**：符号引用 → 直接引用（内存地址）

**初始化**：执行静态代码块和静态变量赋值（`<clinit>` 方法）

---

### 4.2 双亲委派模型

```
Bootstrap ClassLoader（启动类加载器）
        ↑ 父
Extension ClassLoader（扩展类加载器）
        ↑ 父
Application ClassLoader（应用类加载器）
        ↑ 父
  自定义 ClassLoader
```

**加载一个类的流程：**
1. 先让父加载器尝试加载
2. 父加载器无法加载，才由自己加载

**作用：**
- 防止核心类库被篡改（如自定义 `java.lang.String` 会被 Bootstrap 拦截）
- 保证同一个类只被加载一次

**打破双亲委派的场景：**
- JDBC / SPI：启动类加载器加载接口，但实现类在应用层 → 用线程上下文类加载器反向委托
- OSGi、Tomcat：每个模块用独立类加载器，类隔离

## 五、JVM 调优实践

### 5.1 常用 JVM 参数

```bash
# 堆大小（-Xms 初始，-Xmx 最大，建议设成一样，避免动态扩缩容）
-Xms4g -Xmx4g

# 新生代大小（老年代 = 堆 - 新生代）
-Xmn2g

# 元空间（方法区）
-XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=512m

# 选择 GC
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200    # G1 目标停顿时间

# GC 日志（生产必开）
-Xlog:gc*:file=/var/log/gc.log:time,level,tags:filecount=5,filesize=100m

# OOM 时生成 heap dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/heapdump.hprof
```

---

### 5.2 线上问题排查工具

| 工具 | 用途 |
|------|------|
| `jps` | 列出 JVM 进程 |
| `jstat -gc <pid> 1000` | 实时查看 GC 状态、各区内存占用 |
| `jmap -heap <pid>` | 查看堆配置和使用情况 |
| `jmap -dump:format=b,file=heap.hprof <pid>` | 导出 heap dump |
| `jstack <pid>` | 打印线程栈（排查死锁、CPU 100%） |
| Arthas `dashboard` | 实时 JVM 监控面板 |
| Arthas `heapdump` | 在线 heap dump |
| MAT（Memory Analyzer） | 分析 heap dump，找内存泄漏 |

---

### 5.3 CPU 100% 排查步骤

```bash
# 1. 找 Java 进程 PID
jps -l

# 2. 找 CPU 最高的线程（十进制 TID）
top -Hp <pid>

# 3. 十进制转十六进制
printf "%x\n" <tid>

# 4. jstack 找到对应线程的堆栈
jstack <pid> | grep -A 30 "<tid十六进制>"
```

---

### 5.4 内存泄漏排查步骤

```
1. 通过 jstat 观察 Old 区持续增长，Full GC 后回收效果越来越差
2. jmap dump 堆快照
3. MAT 打开 heap dump，查看"Leak Suspects"报告
4. 定位持有引用的对象链路（谁持有了大量对象没释放）
```

常见内存泄漏场景：
- 静态集合无限增长（`static List<Object>`）
- 未关闭的资源（Connection、Stream）
- ThreadLocal 使用后未 remove（配合线程池会导致泄漏）

## 六、面试高频问答

### Q1：JVM 内存结构有哪些区域，分别存什么？

**答：**

JVM 内存分线程私有和线程共享两类：

**线程私有（每个线程独立）：**
- 程序计数器：记录当前执行到哪条字节码指令，是唯一不会 OOM 的区域
- 虚拟机栈：每次方法调用创建一个栈帧，存局部变量、操作数栈等；栈太深抛 StackOverflowError
- 本地方法栈：服务 Native 方法

**线程共享（所有线程共用）：**
- 堆：存所有对象实例，GC 主要工作区域，内存不足抛 OutOfMemoryError: Java heap space
- 方法区（元空间）：存类元数据、常量池，JDK8 改为元空间使用本地内存，不足抛 OutOfMemoryError: Metaspace

---

### Q2：Minor GC / Major GC / Full GC 有什么区别？

**答：**

- **Minor GC**：回收新生代（Eden + Survivor），触发频率高但速度快，因为新生代对象大多数是"朝生夕死"的。
- **Major GC**：回收老年代，通常比 Minor GC 慢 10 倍以上。
- **Full GC**：同时回收堆（新生代 + 老年代）和方法区。触发条件：老年代空间不足、方法区空间不足、调用 System.gc()。Full GC 是停顿最长的，生产上要尽量避免频繁 Full GC。

---

### Q3：什么是 STW（Stop-The-World）？为什么 GC 需要 STW？

**答：**

STW 是 GC 发生时暂停所有用户线程的机制。GC 需要 STW 的核心原因：**保证 GC 期间对象引用关系不变**。

如果 GC 标记存活对象时，用户线程同时在修改引用关系（比如把一个存活对象的引用断开），就会出现"标记漏掉存活对象"（对象消失问题），导致把存活对象当垃圾回收，引发程序错误。

现代 GC（G1、ZGC）通过并发标记 + 屏障技术减少 STW 时间，但无法完全消除。

---

### Q4：双亲委派模型是什么？有什么好处？

**答：**

类加载时，先让父加载器尝试加载，父加载器无法加载再由自己加载，形成层级委派。

好处：
1. **安全性**：防止核心类库被篡改。自定义 `java.lang.String` 会被 Bootstrap ClassLoader 拦截，加载的仍是 JDK 的 String。
2. **避免重复加载**：同一个类只被加载一次，保证全局唯一的 Class 对象。

打破场景：JDBC（SPI 机制）、Tomcat（多应用类隔离）、OSGi（模块化热部署）。

---

### Q5：ThreadLocal 为什么会内存泄漏？怎么解决？

**答：**

ThreadLocal 的 Entry 中 key（ThreadLocal 对象）是弱引用，value 是强引用。

当外部不再持有 ThreadLocal 对象时，key 被 GC 回收变为 null，但 value 仍然被线程的 ThreadLocalMap 强引用，无法回收。

配合线程池使用时更危险：线程不会死，value 会一直留在 ThreadLocalMap 中，越积越多。

**解决方案：使用完后调用 `threadLocal.remove()`**，手动清除 Entry。

---

### Q6：G1 和 CMS 的核心区别是什么？

**答：**

| 对比项 | CMS | G1 |
|--------|-----|-----|
| 作用范围 | 仅老年代 | 全堆 |
| 算法 | 标记-清除（有碎片） | 复制+标记整理（无碎片） |
| 内存碎片 | 有，可能触发 Serial Old 兜底 | 无 |
| 停顿可预测 | 不可预测 | 可设置目标停顿时间 |
| 适合场景 | 4G 以下，低延迟 | 大堆（>4G），可预测停顿 |

JDK 9 起 G1 成为默认 GC，CMS 在 JDK 14 被移除。

---

### Q7：一个对象从创建到被回收，经历了什么？

**答：**

1. **创建**：`new` 时先在 Eden 区分配内存
2. **Minor GC 存活**：Eden 满时触发 Minor GC，存活对象复制到 Survivor，年龄 +1
3. **Survivor 轮换**：每次 Minor GC，S0/S1 互换，对象年龄不断增长
4. **晋升老年代**：年龄超过阈值（默认15）或 Survivor 放不下，晋升老年代
5. **Full GC 回收**：老年代满时触发 Full GC，对象仍不可达则被回收
6. **finalize**：对象被回收前 JVM 可能调用 `finalize()`（不保证时机，生产不要依赖）
